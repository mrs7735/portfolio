The Infostealer Economy: How 1.8 Billion Stolen Credentials Power 2026 Cybercrime

DATE:2026.04.18
READ:20 MIN READ
AUTHOR:CYBER_OPERATIVE
InfostealerLummaStealCSession TheftCookie TheftDBSC2026

In 2025 a single category of malware quietly produced more breach precursors than any other: infostealers. KELA's threat intelligence indexed 3.9 billion compromised credentials from 4.3 million infected devices in 2024, a number that grew dramatically in 2025. SpyCloud's 2025 Identity Exposure Report documented 548 million credentials exfiltrated by stealer malware, with an average of 1,861 cookies harvested per infection. Verizon's 2025 DBIR found that 54% of ransomware victims had their domain credentials appear in stealer log marketplaces before the attack hit. This is the supply chain feeding modern cybercrime — and most security teams are looking the wrong direction.

⚠ DEFENSIVE RESEARCH — All information drawn from publicly available threat intelligence reports: KELA 2025 Threat Intelligence Report, SpyCloud 2025 Identity Exposure Report, Recorded Future March 2026 Identity Trend Report, Verizon 2025 Data Breach Investigations Report, Flashpoint H1 2025 Cyber Threat Report, Constella 2026 Identity Breach Report, Google DBSC technical documentation.

What Infostealers Actually Are

The term "infostealer" obscures what the malware actually does. These are not opportunistic password grabbers. They are highly engineered, single-purpose tools designed to extract everything an authenticated user has access to in their browser, operating system, and applications — then ship it to an attacker-controlled server in a structured format ready for resale or follow-on attack.

# What a typical infostealer collects from a single infected device:
#
# Browser data (from Chrome, Edge, Firefox, Brave, Opera):
# - Saved passwords (decrypted from the OS credential store)
# - Saved credit card numbers and CVVs
# - Autofill data: addresses, phone numbers, names, dates of birth
# - Session cookies for every site the user is logged into
# - Browser history (often valuable for targeting)
# - Bookmarks (reveals interesting internal URLs)
# - Extension data, including some password manager extensions
#
# Operating system credentials:
# - Windows DPAPI-encrypted credentials
# - Stored Wi-Fi passwords
# - Cached domain credentials (from cached logon)
# - SSH keys (known locations checked)
# - VPN configurations
# - RDP saved connections
#
# Cryptocurrency assets:
# - Wallet files (MetaMask, Exodus, Atomic, Electrum, dozens more)
# - Browser extension wallet data
# - Mobile wallet data if synced through cloud
#
# Application data:
# - Discord tokens (instant account takeover)
# - Telegram session files
# - Steam, Epic, Battle.net session tokens
# - FTP client passwords (FileZilla, WinSCP)
# - Email client credentials (Outlook, Thunderbird)
# - Various SaaS app tokens stored locally
#
# System reconnaissance:
# - Running processes (identifies AV/EDR, security tools)
# - Installed software (helps target follow-on exploitation)
# - System specs and timezone
# - Screenshots of desktop and active windows
# - Geolocation via IP

# Output: a single ZIP file ("log") containing all of the above
# Average size: 5-50MB compressed, varies with browser data volume
# This becomes the unit of trade in stealer marketplaces

1. The Top Players in 2025-2026

The infostealer market is not a chaotic landscape of one-off malware. It is a competitive industry with established brands, paid affiliate programmes, customer support channels, and rebranding strategies after law enforcement disruption. Understanding the major families is essential to understanding what your environment is up against.

# Major infostealer families (2025-2026):
#
# Lumma (LummaC2) — Until May 2025 takedown
# Operated as Malware-as-a-Service: $250-$1000/month subscriptions
# Estimated 15-25% of all infostealer infections globally in 2024-early 2025
# Disrupted by Microsoft DCU and law enforcement action May 2025
# Successor operations resumed under modified branding within weeks
#
# StealC / StealC v2 — Active throughout 2025
# Rebranded as "v2" after disruption attempts
# Particularly effective at browser data extraction
# Sold via Russian-language underground forums
# Subscription model around $200-$500/month
#
# RedLine — Mature family, persistent presence
# One of the longest-running infostealer brands
# Modular design: customers pick which collection modules to enable
# Continued to top many infection charts despite age
#
# Acreed (ACR Stealer) — Emerged 2025
# Private operation rather than open MaaS
# Restricted affiliate access; higher technical sophistication
# Includes EDR evasion techniques unusual for the category
#
# Banshee (macOS) — First major macOS infostealer
# Active throughout 2025; brief disappearance October 2025
# Borrowed Apple XProtect signatures to hide from detection
# Confirmed shift in attacker investment priorities
#
# AMOS / Atomic macOS Stealer — macOS market dominant
# Active most of 2025, returned February 2026 after October disruption
# MacSync (formerly Mac.C) became commodity macOS option by year-end 2025
#
# Storm — Emerged early 2026 (under $1000/month)
# Notable architecture: ships encrypted browser credentials to operator's
# server for remote decryption. Avoids loading SQLite libraries on victim
# (which EDR products commonly hook). Lower endpoint footprint.

# The economic picture (per public threat intel):
# Subscription pricing: $150-$1500/month depending on tier
# Stealer log resale on marketplaces: $5-$200 per log depending on contents
# "Premium" logs containing corporate VPN/SSO access: thousands per log
# Marketplace volume: millions of logs traded annually
# Major venues: Russian Market, 2easy, Genesis (until takedown), private Telegram channels

2. The Distribution Pipeline — How Infostealers Reach Victims

Infostealers don't break in. They are invited in. The dominant distribution pattern in 2025-2026 relies on attackers exploiting the search and download habits of normal users.

# Distribution channel 1: Malicious search ads (most common in 2025)
#
# Attacker pattern:
# - Buy Google Ads or Bing Ads for popular software search terms
# - "Notion download", "OBS Studio", "AnyDesk", "Slack download"
# - Ads link to attacker-controlled domain mimicking the real software
# - Domain serves an installer that includes the legitimate software AND the stealer
# - User installs successfully, software works, stealer runs in background
#
# This pattern was so prolific in 2024-2025 that Google rolled out multiple
# advertiser verification changes specifically to address it.

# Distribution channel 2: Cracked software and game cheats
#
# Long-standing infection vector
# Pirated software bundles, Photoshop cracks, game trainers
# Users actively expect to disable antivirus to install — security training fails here
# Disproportionately effective against gamers and home developers

# Distribution channel 3: Fake update prompts
#
# Browser pop-ups: "Your Chrome is out of date, click to update"
# Fake CAPTCHA prompts: "Click here to verify you are human" 
#                       (then "Run this PowerShell command to complete verification")
# Documented as "ClickFix" attack pattern throughout 2025
# Particularly effective because it bypasses email gateway scanning entirely

# Distribution channel 4: Compromised legitimate websites
#
# Attacker compromises a real software vendor's download page
# Real software replaced (or wrapped) with infostealer-bundled version
# Users get the genuine site, real download, infected file
# Detection: hash mismatch with vendor-published checksums

# Distribution channel 5: GitHub and package manager poisoning
#
# Typosquatted npm/PyPI packages with embedded stealer
# Compromised legitimate packages used as dependency
# Compromised GitHub releases (replacing binaries on existing repos)
# CrowdStrike and Sonatype have repeatedly documented this pattern in 2025

# Distribution channel 6: AI-themed lures
#
# Fake "ChatGPT desktop app", "Claude installer", "Midjourney for Windows"
# Capitalises on rapid AI tool adoption and confusion about what's legitimate
# Particularly aggressive growth through 2025 per Recorded Future data

3. The Cookie/Session Theft Problem — Why MFA Doesn't Help

The single most important shift in stealer capability is the move from password theft to session token theft. Recorded Future's March 2026 report documented 276 million credentials exposed with cookies — 31% of all malware-sourced credentials. December 2025 was the highest single month for cookie exposure in the index. Understanding why this matters explains why MFA is no longer sufficient protection.

# What a session cookie actually is:
#
# When you log into a website with username + password + MFA, the server
# issues a session cookie. This cookie is your "I'm logged in" badge for
# subsequent requests. Each request your browser makes to the site includes
# the cookie, which the server validates without re-checking password or MFA.
#
# Typical session cookie names:
# - JSESSIONID, PHPSESSID (generic)
# - sessionid (Django, Flask)
# - .ASPXAUTH (ASP.NET)
# - ESTSAUTH, ESTSAUTHPERSISTENT (Microsoft Entra)
# - SAPISID, SSID, APISID (Google)
# - sid, refresh_token (modern OAuth-based apps)
#
# The cookie may be valid for hours, days, or even months depending on the site.
# Banking: typically 15-30 minutes
# Microsoft 365 / Google Workspace: weeks
# Most consumer SaaS: days to months

# How session theft (also called "pass-the-cookie") works:
#
# Step 1: Infostealer extracts session cookies from victim's browser
#         (browsers store these in SQLite databases; OS credential store
#          encrypts the master key, but a process running as the user
#          can decrypt them — this is by design for usability)
#
# Step 2: Cookies appear in stealer logs sold on marketplaces
#         Many marketplaces let buyers search by site:
#         "Find logs containing valid Microsoft 365 cookies for [target domain]"
#
# Step 3: Buyer imports cookies into their own browser
#         Browser extensions (Cookie-Editor, EditThisCookie) make this trivial
#         Buyer navigates to the target site
#         Site validates the cookie → buyer is logged in as the victim
#         No password required. No MFA required. The session is already authenticated.

# Why MFA cannot stop this:
# MFA protects the LOGIN event. The cookie is issued AFTER successful login.
# Once the cookie exists, the system has already verified MFA.
# Replaying the cookie produces no new login event — therefore no new MFA challenge.
# This is not an MFA bypass; the attacker simply skipped the step entirely.

4. The Path from Stealer Log to Ransomware

The 54% statistic from Verizon DBIR 2025 — "54% of ransomware victims had their domain credentials in stealer log marketplaces before the attack" — describes a specific operational pipeline. Initial Access Brokers (IABs) buy stealer logs, sift them for corporate value, and resell access to ransomware operators. Time from infection to ransomware encryption can be under 48 hours.

# The stealer-to-ransomware pipeline:
#
# Day 0: Employee at Corp downloads "free PDF converter" from search ad
# Stealer infection occurs. Stealer log uploaded to operator's panel.
# Log contains: VPN credentials, Microsoft 365 cookies, Confluence access
#
# Day 0-3: Stealer operator's automated tools tag the log
# "Domain: corp.com (Fortune 1000) - HIGH VALUE"
# Log placed in premium-tier listing on marketplace
#
# Day 3-7: Initial Access Broker buys the log for $500-$2000
# IAB tests the credentials, confirms they work
# IAB packages "VPN access to Corp" as a sellable asset
#
# Day 7-14: Ransomware affiliate buys access from IAB for $5,000-$50,000
# Affiliate uses VPN credentials to reach internal network
# Internal reconnaissance: Active Directory, file servers, hypervisors
# Lateral movement and privilege escalation (independent skills)
# Backup destruction
# Mass encryption and ransom demand
#
# This pipeline means defenders have a clear early warning signal:
# IF you can detect that a stealer log containing your corporate credentials
# has appeared on a marketplace, you have days to weeks before ransomware
# may follow. Many vendors offer this monitoring service as
# "stealer log surveillance" or "compromised credential monitoring."

# Why traditional credential monitoring misses this:
# Old credential breach monitoring: looks for password hashes from old breaches
# Modern infostealer monitoring: looks for fresh logs containing
# - Username + password + URL combinations
# - Session cookies for specific high-value sites
# - Cryptographic identifiers tied to your domain

5. Defences That Actually Work in 2026

Conventional credential security advice (password complexity, MFA, periodic rotation) does not address the infostealer threat model. Effective defences require a combination of endpoint hardening to prevent the initial infection, infrastructure changes to limit damage from a successful infection, and monitoring to detect compromised credentials in marketplaces.

# Defence 1: Application allowlisting on endpoints (highest impact)
#
# Most stealer infections require user download and execution
# Application allowlisting (WDAC, AppLocker) blocks unknown executables
# Trade-off: significant operational overhead for IT
# Reality: not deployed in most enterprises despite being the strongest control

# Defence 2: Browser hardening
#
# Chrome's Device Bound Session Credentials (DBSC) — rolled out late 2025
# Cryptographically binds session cookies to the device's TPM
# Stolen cookies cannot be replayed from another device
# Enable on managed devices: chrome://flags/#enable-bound-session-credentials
# Enterprise policy: BoundSessionCredentialsEnabled = true
#
# Microsoft Edge has equivalent: requires Windows 11 with TPM 2.0
# Firefox: similar capability in development
#
# DBSC is the single most impactful 2025-2026 defensive shift for cookie theft

# Defence 3: Short token lifetimes for high-value applications
#
# Microsoft 365 Conditional Access: configure shorter session lifetimes
# Sensitive applications (admin consoles, finance tools): 1-4 hour sessions
# Force re-authentication for high-risk activities (admin actions, downloads)
# Trade-off: user friction; needs balanced configuration

# Defence 4: Conditional Access policies based on device compliance
#
# Require sessions to originate from a managed, compliant device
# Stolen cookies replayed from attacker's machine fail device check
# Microsoft Entra Conditional Access: "Require Hybrid Azure AD Joined device"
# Combined with token binding: significant raise of the bar

# Defence 5: Monitoring for your domain in stealer marketplaces
#
# Commercial services (SpyCloud, Recorded Future, Constella, Flare) monitor
# infostealer marketplaces and alert when credentials matching your domain appear
# Free option: Have I Been Pwned now includes infostealer log corpus
# When alert fires: rotate password AND invalidate all sessions for that user

# Defence 6: Just-in-time privilege elevation
#
# Standing admin access multiplies stealer log impact
# Privileged Access Management (PAM) tools issue admin credentials only when needed
# Even if a workstation is compromised, no permanent admin creds to steal

# Defence 7: User education that addresses real distribution channels
#
# Old advice "don't click suspicious links" is too vague to act on
# Specific 2026 advice that helps:
# - Always type software download URLs directly; never click search ads
# - For software, verify the publisher's exact domain (slack.com not slack-download.com)
# - Beware "fake CAPTCHA" / "click to verify" prompts that ask for PowerShell
# - Cracked software and game cheats are the highest-risk category

6. What to Do Immediately

# Immediate action checklist for security teams:
#
# ☐ Subscribe to a stealer log monitoring service (SpyCloud, Recorded Future,
#   Constella, Flare, or equivalent)
#   Get visibility into which of your credentials are already on marketplaces
#
# ☐ Enable Chrome DBSC on all managed devices
#   This is the single most impactful change for cookie-based attacks
#
# ☐ Review session token lifetimes for sensitive applications
#   Conditional Access in Entra ID, equivalent in Okta, etc.
#   Lower limits for: admin consoles, finance, HR, source code platforms
#
# ☐ Implement Conditional Access policies requiring device compliance
#   Combined with DBSC: stolen cookies become much harder to weaponise
#
# ☐ Deploy application allowlisting where feasible
#   Start with high-risk roles (developers, executives, finance)
#   Expand as operational maturity grows
#
# ☐ Review browser extension allowlist
#   Many infostealers are delivered as malicious browser extensions
#   Restrict extension installation to a vetted allowlist
#
# ☐ Educate users on the specific 2026 distribution patterns
#   Search ad redirects, fake CAPTCHA / ClickFix prompts, cracked software
#   Generic phishing training does not cover these patterns
#
# ☐ Tabletop exercise the stealer-to-ransomware scenario
#   "Employee credentials and session cookies appeared in a marketplace"
#   Walk through detection, containment, eradication, recovery
#   Identify gaps in your current playbook

The Strategic Picture

Infostealers shifted the economics of cybercrime in a way that conventional defences are not yet adapted to. The attacker no longer needs to compromise an enterprise. They need to compromise an employee's home computer, harvest browser data, and sell the result to someone with the operational skill to weaponise it. This decoupling of initial access from monetisation has created a high-volume, low-skill input feeding an ecosystem of more sophisticated downstream attackers.

The defensive implication is that the perimeter has dissolved. The compromised endpoint may be a home computer your security team will never see. The compromised credentials are bought and sold by people you cannot identify. The fastest path to ransomware in 2026 increasingly starts with a stealer log marketplace, not a phishing email or a network exploit. Building defences that assume credential compromise has already happened — and limiting what an attacker can do with stolen credentials — is the only durable answer.
